1 概述
1.1 编写目的
为统一美食在线商城系统测试执行标准，规范测试用例编写、缺陷定义与提交、回归测试、文档管理、版本协作等全流程工作，保证测试过程标准化、结果可追溯，保障测试质量与工作效率，特制定本规范。
本规范适用于本项目所有测试工作，全体项目成员需遵照执行。
1.2 适用范围
本规范覆盖项目前台用户端、后台管理端所有功能测试、权限测试、UI 测试、兼容性测试、回归测试；同时约束测试文档编写、缺陷管理、Git 文档提交等相关工作。

2 测试用例编写规范
2.1 用例基本结构
每条测试用例必须包含以下字段，顺序保持统一：
用例编号
所属模块
用例名称
前置条件
测试步骤
预期结果
用例优先级
2.2 编号命名规则
统一格式：TC-模块缩写-三位序号
示例：
TC-Login-001 登录模块第 1 条用例
TC-Goods-001 商品模块第 1 条用例
TC-Order-001 订单模块第 1 条用例
2.3 场景覆盖要求
核心功能必须覆盖正向流程、反向流程、边界值、异常场景、越权场景；
业务规则类功能（库存、订单、支付、权限、内容审核）需单独设计专项用例；
步骤描述简洁清晰，一步一操作，具备可复现性；
预期结果明确、具体，与需求及业务规则保持一致，不使用模糊描述。
2.4 优先级定义
P0（最高）：核心业务流程、系统基础功能，影响整体使用，必测项
P1（高）：主要功能模块，功能异常会影响大部分用户使用
P2（中）：次要功能、界面交互类功能
P3（低）：体验优化、文案美化、界面建议类场景

3 缺陷管理规范
3.1 缺陷等级划分（四级标准）
3.1.1 P0 致命缺陷
定义：系统崩溃、核心功能完全不可用、数据错乱、数据丢失、库存超卖、重复支付、权限严重泄露等阻断性问题。
要求：立即修复，优先处理，未修复不得进入下一测试阶段。
3.1.2 P1 严重缺陷
定义：主要业务流程中断、关键页面无法访问、业务规则失效、重要功能逻辑错误。
要求：本轮迭代必须修复，修复完成后立即复测。
3.1.3 P2 一般缺陷
定义：页面样式错乱、图文排版异常、提示文案错误、次要交互异常、非核心功能小问题。
要求：可根据项目进度合理安排修复，不影响核心业务即可暂缓。
3.1.4 P3 建议类缺陷
定义：操作体验优化、界面美化、文案润色、布局微调等优化类意见。
要求：非强制修复，可纳入后续版本迭代优化。
3.2 缺陷编号规则
统一格式：BUG-三位序号
示例：BUG-001、BUG-002
3.3 缺陷内容必填项
提交缺陷时，信息必须完整，禁止模糊描述：
缺陷编号
所属功能模块
缺陷简要标题
详细复现步骤（步骤完整、可 100% 复现）
实际运行结果
预期正确结果
缺陷等级
问题归属（前端开发/后端开发）
缺陷状态
佐证截图（UI、报错、数据异常类必须附截图）
3.4 缺陷状态流转
状态分为：待修复、修复中、已修复、复测不通过、已关闭、暂缓修复
新发现缺陷：标记为「待修复」，指派对应开发人员；
开发接收并处理：标记为「修复中」；
开发完成修复提交版本：标记为「已修复」，由 QA 执行复测；
复测问题依旧：标记为「复测不通过」，退回开发重新修复；
复测验证通过：标记为「已关闭」；
经团队确认暂不修复：标记为「暂缓修复」，备注原因。

4 测试执行规范
4.1 通用执行要求
严格按照测试用例逐条执行，不跳项、不漏项；
执行过程如实记录实际结果，与预期不一致立即提交缺陷；
测试过程保持环境纯净，不随意修改数据库、系统配置；
发现隐性问题（用例未覆盖但存在异常），同步新增缺陷并补充测试用例。
4.2 功能测试要求
严格遵循项目业务规则，重点校验库存控制、订单状态、支付拦截、商品上下架、内容审核等核心逻辑；
区分游客、普通用户、商品管理员、超级管理员四类身份，逐一对权限进行验证。
4.3 权限测试要求
针对不同角色开展越权访问测试，禁止跨角色操作；
验证菜单、按钮、页面 URL 访问权限，无权限角色必须被拦截；
权限变更后，及时复测权限有效性。
4.4 兼容性测试要求
固定测试浏览器：Chrome、Edge、Firefox 三款主流浏览器；
每轮版本更新，均需重新执行基础兼容性校验；
页面布局、按钮点击、表单提交、图片展示为重点校验项。

5 回归测试规范
5.1 回归测试触发条件
开发完成缺陷修复，提交新版本；
项目新增功能、修改原有业务逻辑；
数据库结构、权限配置、核心代码发生变更。
5.2 回归测试范围
必测：所有本次已修复的缺陷；
选测：缺陷关联的周边功能、同一模块核心流程（增量回归）；
版本大幅更新：执行全量用例回归（全量回归）。
5.3 回归执行要求
按缺陷清单逐条复测，记录复测结果；
复测不通过，详细注明问题原因，退回开发；
复测通过，关闭对应缺陷；
重点模块（订单、支付、库存、权限）回归力度加强，避免重复问题。

6 文档管理规范
6.1 文档存放目录
严格按照项目仓库目录划分，不得随意存放：
docs/qa/        // 测试计划、测试规范
tests/          // 测试用例、缺陷清单、各类测试报告
6.2 文档格式要求
统一使用 Markdown / Excel 格式，排版整洁、层级清晰；
文档名称固定，不随意修改文件名；
内容修改后及时更新版本与日期，保证线上文档为最新版。
6.3 文档更新要求
需求变更、功能调整后，24 小时内同步更新对应测试用例与文档；
每一轮测试结束后，及时汇总缺陷数据、更新台账。

7 Git 协作提交规范
7.1 分支命名规则
格式：feature/QA/功能描述
示例：
feature/QA/test-plan （提交测试计划）
feature/QA/test-case （提交测试用例）
feature/QA/bug-record （提交缺陷清单）
feature/QA/test-report （提交测试报告）
7.2 Commit 提交备注
统一前缀：[QA] + 操作内容
示例：
[QA] 编写并提交测试规范文档
[QA] 新增全模块功能测试用例
[QA] 更新缺陷清单及状态
[QA] 提交回归测试报告
7.3 提交原则
禁止直接在 develop、main 主干分支编写 / 提交文档；
所有修改从主干分支拉取新分支，完成后提交 PR 审核；
单次提交内容单一，不混合多种类型文件。
